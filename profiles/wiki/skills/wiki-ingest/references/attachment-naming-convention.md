# Attachment naming convention

Use when archiving or renaming binary/source sidecar artifacts in `/vault/Attachments/`.

## Folder policy

Keep binary artifacts in `/vault/Attachments/`; do **not** move the whole Attachments folder into `/vault/Sources/`.

- `Sources/` = Markdown/text source provenance and article extracts.
- `Attachments/` = binary/sidecar provenance: PDFs, images, screenshots, thumbnails, captions, `.info.json`, `.srt`, etc.

This keeps source scanners, source filename normalization, wiki ingest, and link checks from mixing Markdown archives with binary files.

## Canonical filename pattern

For source-aligned standalone attachments, prefer:

```text
YYYY-MM-DD - Source Title - Attachment Descriptor.ext
```

Examples:

```text
2026-07-06 - The Economist USA - May 30th-June 5th 2026 - Cleaned No Ads.pdf
2026-06-22 - VSOCC Infant Food Exposure Checklist Page 1.jpg
2026-06-28 - Lymphatic Drainage Node Order Screenshot.jpg
```

Use the source ingest date/title when the attachment clearly belongs to a normalized source archive. Use a concise descriptor such as `Cleaned No Ads`, `Page 1`, `Thumbnail`, or `Transcript`.

## Link migration after rename

When renaming an attachment, rewrite both pathful and bare Obsidian references across writable vault areas:

```md
attachment: Attachments/old-name.pdf
![[Attachments/old-name.jpg]]
![[old-name.jpg]]
[[old-name.pdf]]
```

into the new attachment filename. Verify that content search finds zero references to the old names.

## What to skip

- Do not rename legacy `Attachment-YYYY...jpg` files if they are heavily referenced from read-only `Daily/` notes unless you are explicitly doing a vault-wide link migration that includes read-only areas.
- Do not flatten or rename every `yt-dlp` sidecar file. Keep social/video metadata folders dated/descriptive, and keep sidecar names generated by the extractor unless there is a clear readability problem.
- Do not store video/audio downloads as an ingest workaround; preserve metadata/transcripts/thumbnails only.

## Verification checklist

After an attachment normalization pass:

1. Expected new attachment files exist.
2. Removed junk files such as `.DS_Store` if present.
3. Old renamed attachment references return zero matches.
4. Representative source archives and Notes show the new attachment names.
5. `wiki-log.md` records the migration.
6. `wiki-ingest`/vault policy still says Attachments stays separate from Sources.
