# Telegram Task Button Drip Pattern

Use this when a user wants actionable Telegram cards with inline buttons, especially when the source of truth is Obsidian task files.

## Shape

- Use a `no_agent=True` script-only cron job for the sender.
- Schedule the cron as a drip, e.g. `*/10 8-23 * * *` for one card every 10 minutes from 8:00AM through 11:50PM local scheduler time.
- The sender script should send **at most one card per run**. Use an env-configurable limit such as `TASK_BUTTON_SEND_LIMIT`, default `1`.
- Keep stdout empty on success so script-only cron stays silent unless there is an error.

## Source of truth

Prefer reading `/vault/Tasks/tasks/*.md` directly instead of depending on a sidecar generated by a previous briefing job. Parse YAML frontmatter:

```yaml
type: task
due_date: YYYY-MM-DD
status: pending | in_progress | completed | blocked
```

Include tasks where:

- `type: task` or missing type treated as task
- `status` is not `completed`, `done`, `cancelled`, or `canceled`
- `due_date <= today`

## Ordering

User preference for task cards:

1. Due today first
2. Overdue tasks after today
3. For overdue tasks, newest overdue first is usually less stale/noisy than oldest-first

## Duplicate suppression

Maintain a registry such as `~/.hermes/profiles/<profile>/task_buttons/registry.json` keyed by a digest of date/title/file path. Store:

```json
{
  "id": "digest",
  "date": "YYYY-MM-DD",
  "title": "Task title",
  "file_path": "/vault/Tasks/tasks/example.md",
  "status": "sent",
  "message_id": 123
}
```

Skip registry entries with `status: sent` or `status: done` so the same task is not resent repeatedly. If the user deletes a task, mark matching registry entries `deleted` to suppress stale resend.

## Button callback

The Telegram gateway callback can use compact fixed callback data like Thor wellness buttons:

- `wl:water:500` for wellness logs
- `zt:<task_id>` for Zeus task Done buttons

On `zt:<task_id>`:

1. Look up the task in the registry.
2. Verify the caller is authorized.
3. If `file_path` starts with `/vault/Tasks/tasks/`, update the source Markdown frontmatter to `status: completed`.
4. Mark registry entry `done` with `done_at` and `done_by`.
5. Edit the Telegram message to `✅ Done — <title>` and remove the inline keyboard.

## Pitfalls

- Do not send the whole backlog in one cron tick. A large vault can spam dozens of cards. Enforce max-one-card-per-run before scheduling every 10 minutes.
- Do not rely on an LLM briefing job to generate the sidecar if the user wants direct task cards. Direct vault parsing is more deterministic and avoids failures from upstream agent timeouts.
- When changing schedules, always `cronjob(action="list")` first, then update by the returned job ID.
- Remember cron schedules use the scheduler's configured timezone/local interpretation. For Vancouver users, phrase replies as Pacific/local clock time unless they explicitly request true PST/UTC.
